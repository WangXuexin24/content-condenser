# Content Condenser

三种模式压缩内容。选对的，不选贵的。

## 选型指南

```
输入是什么？ ── 机器输出（日志、CLI结果） ──→ L1 字符级（机械，<10ms）
            │
            ── 人类写的文档、对话历史 ──→ L2 语义级（crystalize，仍可读）
            │
            ── 喂给AI的上下文、子代理通讯 ──→ L3 极限级（compress-for-ai，仅AI可读）
```

| 内容类型 | 用哪个 | 预期节省 | 代价 |
|---------|--------|---------|------|
| CLI 输出、CMake 日志、npm install | L1 | 10-20% | 无 |
| 文档、对话、wiki、教程 | L2 | 40-70% | 一次 AI 判断 |
| memory/ 存档、子代理上下文 | L2→L3 连用 | 60-80% | AI判断+丢失可读性 |
| 代码（不压缩源文件） | 别用 | — | — |

---

## L1：字符级压缩（机械，无 AI 调用）

**脚本：** `scripts/strip_chars.py` — 零依赖纯 Python

原理：LLM 能理解残缺文本。"Ths fnctn tks intgrs" → 人类需要猜，AI 直接懂。

```bash
cat log.txt | python3 scripts/strip_chars.py -l 2 -s
python3 scripts/strip_chars.py doc.txt -l 3 -m
```

**四级深度：**
- `-l 1` 去重复字母（~1-2% 节省，完全可读）
- `-l 2` +去元音（~10-16% 节省，大致可读）
- `-l 3` +截断长词（~13-17% 节省）
- `-l 4` +去填充词+去重行（~15-20% 节省）

**自动保护：** URL、邮箱、UUID、代码块(```)、电话号码。

**适用：** CLI 工具输出、日志、构建结果。

---

## L2：语义压缩（crystalize，AI 判断）

**目标：** 删无用句，保留人类可读。

**流程：**
1. 明确文档核心信息
2. 逐句评估：删掉这句，核心信息还完整？
3. 保留：事实/数据/逻辑/结论。删：铺垫/过渡/重复
4. 输出精简版

---

## L3：极限压缩（compress-for-ai，仅 AI 可读）

**目标：** 极限信息密度，人类几乎不可读。

**流程：**
1. 删：礼貌语、模糊限定、过渡词、重复、额外换行
2. 留：事实、逻辑、数据、关系
3. 用：缩写、符号、YAML/JSON 结构代替散文
4. 确保 AI 能重建全部核心信息

**效果示例（286字→68字，省76%）：**
```
项目进度:
- 核心功能: done (dev+test)
- 性能: DB查询 高并发 响应波动 → P0
- UI: 交互体验 待打磨
- ETA: 下月中旬 v1
```

---

## 🚫 永远不要压缩的内容

- 测试失败输出 — 完整 stack trace 才能定位
- 构建错误 — 完整错误上下文映射到修复
- 安全发现 — 敏感信息不可丢失
- 精确 diff — diff 本身就是审查面
- 用户引用的原文 — 原文 = 准确答案
- 代码源文件 — 不用字符级碰代码
- 配置文件 — YAML/JSON/TOML 保持原样

---

## 实用工作流

- **文档存档：** 原始 → L2 crystalize → memory/
- **日志分析：** 500行日志 → L1 字符级 → L2 crystalize → 喂AI
- **子代理通讯：** 任务描述 → L3 compress → 发给子代理
- **对话历史溢出：** 前N轮 → L3 compress | 最新2轮 → 保留原样
- **读大型文档前：** 大文件 → `strip_chars.py -l 2 -m` → 大幅减读取 token
